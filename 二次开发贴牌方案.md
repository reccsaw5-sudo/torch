# Hermes-Agent 二次开发（贴牌 / 锁 URL / 登录注册）方案

> 本文只做架构说明与改造方案，**不含任何代码改动**。
> 所有引用的文件路径均为仓库根目录相对路径。

---

## 一、项目是什么

Hermes-Agent 是一个"同一套 Agent 内核 + 多种前端外壳"的个人 AI Agent 框架。同一个内核跑在：

- **CLI**（`cli.py`，交互式命令行）
- **TUI**（`ui-tui/` Ink + React 终端 UI + `tui_gateway/` Python 后端）
- **Web 仪表盘**（`web/` 前端 + `hermes_cli/web_server.py` FastAPI 后端，`hermes dashboard` / `hermes serve`）
- **桌面 App**（`apps/desktop/`，Electron + React，走 JSON-RPC 连后端）
- **消息网关**（`gateway/`，Telegram / Discord / Slack 等 ~20 个平台）

它通过 **插件（plugins/）和技能（skills/）** 扩展，而不是往内核里加东西。

### 核心文件（你二次开发最常碰的）

| 文件 | 作用 |
|------|------|
| `run_agent.py` | `AIAgent` 类，对话主循环（模型调用、base_url 落地都在这里） |
| `model_tools.py` | 工具编排、工具发现、function call 分发 |
| `toolsets.py` | 工具集定义 |
| `cli.py` | `HermesCLI`，交互式 CLI（模型配置读取、`/model` 等命令） |
| `hermes_cli/config.py` | `DEFAULT_CONFIG`（config.yaml 的默认值）、配置加载 |
| `hermes_cli/skin_engine.py` | **皮肤/品牌引擎**（贴牌关键） |
| `hermes_cli/banner.py` | 启动横幅、Logo |
| `providers/` + `plugins/model-providers/` | 模型供应商 profile（base_url 声明的地方） |
| `hermes_cli/web_server.py` | Web/桌面后端、鉴权中间件 |
| `hermes_cli/dashboard_auth/` | **登录鉴权框架**（登录注册关键） |
| `plugins/dashboard_auth/` | 内置鉴权 provider（basic 密码 / nous OAuth / self_hosted / drain） |

### 三个数据/配置落点

- `~/.hermes/config.yaml` —— 所有行为配置（非机密）
- `~/.hermes/.env` —— **只放机密**（API key / token / 密码）
- `~/.hermes/` 下的 sessions / skills / plugins / logs 等状态

> 重要：代码里读取 HOME 目录一律用 `get_hermes_home()`（`hermes_constants.py`），不要硬编码 `~/.hermes`，否则会破坏"多 profile 多实例"。

## 二、贴牌（换品牌名 / Logo / 配色 / 文案）

Hermes 已内置一套**数据驱动的皮肤引擎**，贴牌**不需要改内核代码**，只写一份皮肤数据即可。

### 皮肤能改什么（`hermes_cli/skin_engine.py`）

| 元素 | 皮肤字段 | 生效位置 |
|------|----------|----------|
| Agent 名称（横幅标题、状态显示） | `branding.agent_name` | `banner.py` / `cli.py` |
| 欢迎语 | `branding.welcome` | `cli.py` |
| 回复框标签 | `branding.response_label` | `cli.py` |
| 输入提示符 | `branding.prompt_symbol` | `cli.py` |
| **ASCII Logo** | `banner_logo` | `banner.py`（替换 `HERMES_AGENT_LOGO`） |
| 横幅边框/标题/强调/正文色 | `colors.banner_*` | `banner.py` |
| 回复框边框色 | `colors.response_border` | `cli.py` |
| Spinner 表情/动词/翅膀 | `spinner.*` | `agent/display.py` |
| 工具输出前缀、每个工具的 emoji | `tool_prefix` / `tool_emojis` | `display.py` |

### 做法（两种，选其一）

**A. 用户级皮肤文件（最轻，零代码）**
在 `~/.hermes/skins/<brand>.yaml` 放一份皮肤：

```yaml
name: mybrand
description: 我的品牌皮肤
colors:
  banner_border: "#0A84FF"
  banner_title: "#FFFFFF"
  banner_accent: "#0A84FF"
banner_logo: |
  [bold #0A84FF]（你的 ASCII 艺术字 Logo，Rich markup）[/]
branding:
  agent_name: "MyBrand Agent"
  welcome: "欢迎使用 MyBrand！输入消息或 /help。"
  response_label: " ◆ MyBrand "
  prompt_symbol: "› "
spinner:
  thinking_verbs: ["思考中", "分析中", "处理中"]
tool_prefix: "▏"
```
然后在 `config.yaml` 里设 `display.skin: mybrand`，或运行时 `/skin mybrand`。
参考现成内置皮肤：`_BUILTIN_SKINS` 里的 `ares` / `slate` / `mono`。

**B. 内置皮肤（随包发行，让贴牌成为默认）**
在 `hermes_cli/skin_engine.py` 的 `_BUILTIN_SKINS` 里加一个键（结构同上），并把 `hermes_cli/config.py` 的 `DEFAULT_CONFIG["display"]["skin"]` 默认值改成你的品牌皮肤名。这样新用户开箱即是你的品牌。

### 皮肤覆盖不到的地方（如果要彻底贴牌，需另外处理）

皮肤引擎目前只覆盖 CLI/TUI 的视觉。以下 UI 的品牌需要单独改：

- **Web 仪表盘品牌**：`web/src/`（Logo、标题、favicon、`web/index.html`、`web/public/`）。
- **桌面 App 品牌**：`apps/desktop/`（`index.html`、`electron/`、`assets/`、`package.json` 里的 productName、图标）。
- **README / 文档站**：`README*.md`、`website/`（如需对外发布）。
- **包名 / 命令名**：命令是 `hermes`（`hermes` 可执行脚本 + `pyproject.toml`）。若要改成 `mybrand` 命令，需改 `pyproject.toml` 的 `[project.scripts]` 与相关引用——这个改动面较大，通常**不建议**在首版就做，保留 `hermes` 命令、只换视觉品牌是性价比最高的方案。

## 三、锁死模型配置 URL（用户不能改 base_url）

### 现状：base_url 的解析链

1. **供应商 profile**：`providers/base.py` 的 `ProviderProfile.base_url` 声明每个供应商的默认端点；具体供应商在 `plugins/model-providers/<name>/__init__.py` 里 `register_provider(...)` 注册。
2. **解析优先级**（`hermes_cli/providers.py::resolve_provider_full`）：
   `config.yaml 里 providers.<name>`（用户自定义，最高）→ 内置 profile → models.dev → …
3. **运行时落地**（`run_agent.py`，`AIAgent`）：
   ```python
   self.base_url = base_url or CLI_CONFIG["model"].get("base_url", "") or ...
   ```
   即：**构造参数 base_url → config.yaml 的 `model.base_url` → 供应商默认**。
4. **用户能改 URL 的入口**（要锁死就得堵住这些）：
   - `config.yaml` 的 `model.base_url`、`providers.<name>`、`custom_providers`
   - CLI 的 `/model` 命令 + 自定义端点向导 `hermes_cli/model_setup_flows.py`（`base_url = input(...)`，约 749、1036 行）
   - `hermes setup` 向导、`.env` 里的相关变量
   - Web/桌面的"设置 → 模型"页

### 推荐方案：**自定义品牌供应商 + 强制覆盖 + 关闭改 URL 的入口**（三层，纵深防御）

**第 1 层 —— 固定一个品牌供应商 profile（源头锁定）**
新建 `plugins/model-providers/mybrand/__init__.py`：

```python
from providers import register_provider
from providers.base import ProviderProfile

register_provider(ProviderProfile(
    name="mybrand",
    base_url="https://api.mybrand.com/v1",   # 锁死的端点
    api_mode="chat_completions",
    env_vars=("MYBRAND_API_KEY",),           # key 仍可来自 .env
    fallback_models=("mybrand-model-1",),
))
```
并把 `hermes_cli/config.py` 的 `DEFAULT_CONFIG["model"]` 相关默认设为该供应商（`provider: mybrand`，不给 `base_url` 留可改口子）。

**第 2 层 —— 在 `AIAgent` 构造后强制覆盖（运行时兜底，最硬）**
在 `run_agent.py` 里 `self.base_url = ...` 落地之后，无条件把它钉死为品牌 URL（忽略任何传入/config 值）。这是"锁死"最可靠的一层——即使用户改了 config.yaml 也无效。同样处理 `delegation.base_url`、`auxiliary.*.base_url`（子 Agent / 辅助 LLM 的端点），否则会有旁路。

> 建议用一个常量（如 `LOCKED_BASE_URL`）集中定义，避免散落。

**第 3 层 —— 移除/禁用改 URL 的 UI 入口（防误导）**
- 关闭 `/model` 的"自定义端点"流程与 `model_setup_flows.py` 的 URL 输入。
- Web/桌面设置页隐藏"模型 base_url / 自定义供应商"编辑项（`web/src/`、`apps/desktop/src/app/settings/`）。
- `hermes setup` 向导跳过 URL 询问。

> 只做第 3 层是"藏起来"，不是"锁死"；**第 2 层是真正的锁**。三层一起做最稳。

> 关于 config：按项目规范，URL 这类非机密配置应放 `config.yaml` 而不是新增 `HERMES_*` 环境变量；但"锁死"场景更适合直接在代码里用常量固定，而不是留在可编辑的 config 里。

## 四、登录 / 注册

### 现状：已有完整的鉴权框架（`hermes_cli/dashboard_auth/`）

面向 **Web 仪表盘 / 桌面 App / `hermes serve`** 的后端已有一套可插拔鉴权：

- **抽象层** `dashboard_auth/base.py`：`DashboardAuthProvider` ABC，定义了三种能力：
  - OAuth 跳转登录：`start_login` / `complete_login`
  - **用户名密码登录**：`supports_password=True` + `complete_password_login`
  - 机器令牌：`supports_token=True` + `verify_token`
  - 会话生命周期：`verify_session` / `refresh_session` / `revoke_session`（HttpOnly Cookie 会话）
- **注册表** `dashboard_auth/registry.py`：`register_provider(...)`（插件在启动时通过 `ctx.register_dashboard_auth_provider(...)` 注册）。
- **中间件/路由/登录页/Cookie**：`middleware.py`、`routes.py`、`login_page.py`、`cookies.py`、`ws_tickets.py`。
- **鉴权门何时开启**（`web_server.py::should_require_auth`）：
  - 绑定到 **loopback（127.0.0.1/localhost）** → 不强制登录（本机可信）。
  - 绑定到**非 loopback（对外/0.0.0.0/局域网）** → **强制**必须有鉴权 provider（OAuth 或密码），`--insecure` 已不能绕过。

### 内置 provider（`plugins/dashboard_auth/`）

| provider | 说明 |
|----------|------|
| `basic` | **单用户名 + 密码**（scrypt 哈希，HMAC 无状态会话，零依赖）。配 `config.yaml: dashboard.basic_auth.{username,password/password_hash,secret}` |
| `nous` | Nous 官方 OAuth |
| `self_hosted` | 自建 OAuth IDP |
| `drain` | 仅机器令牌（非交互） |

### 你的两种典型需求

**需求 1：只要一个"登录密码"保护面板（最简单，几乎零开发）**
直接用内置 `basic` provider：

```yaml
# config.yaml
dashboard:
  basic_auth:
    username: admin
    password_hash: "scrypt$..."   # 用 hash_password() 预生成，避免明文
    secret: "<32+ 字节随机，base64/hex>"
    session_ttl_seconds: 43200
```
生成 hash：
```bash
python -c "from plugins.dashboard_auth.basic import hash_password; print(hash_password('你的密码'))"
```
非 loopback 绑定时登录页会自动渲染账号密码表单。**注意：`basic` 只支持单账号，没有"注册"。**

**需求 2：真正的多用户"登录 + 注册"（需要开发一个鉴权插件）**
`basic` 是单账号、无注册、无用户库。要"注册"就自己实现一个 `DashboardAuthProvider` 插件（**这是官方推荐的扩展点，不用改内核**）：

1. 新建插件 `~/.hermes/plugins/<brand-auth>/`（或仓库内 `plugins/dashboard_auth/<brand>/`），实现类继承 `DashboardAuthProvider`：
   - `supports_password = True`
   - `complete_password_login(username, password)`：查你的**用户数据库**（SQLite/Postgres/…），校验密码（用 `hashlib.scrypt`，参考 `basic` 里的 `hash_password`/`_verify_password`），成功返回 `Session`。
   - `verify_session` / `refresh_session` / `revoke_session`：可直接照抄 `basic` 的 HMAC 无状态会话实现。
2. **注册接口**（"注册新用户"）：框架的登录页只有登录表单，没有注册表单。注册需要你**自己加一个 HTTP 路由**接收 `{username, password, email}` → 写入用户库（scrypt 存哈希）。落点二选一：
   - 在你的插件里用 `ctx.register_...`（若插件上下文暴露了加路由的能力）挂一个 `/auth/register`；
   - 或在 `hermes_cli/web_server.py` 挂一个注册路由（改内核，面小）。
   - 前端注册页：Web 在 `web/src/pages/`，桌面在 `apps/desktop/src/app/`；也可复用/扩展 `dashboard_auth/login_page.py` 的服务端渲染登录页，加一个"去注册"链接。
3. 用 `ctx.register_dashboard_auth_provider(provider)` 注册你的 provider（参考 `plugins/dashboard_auth/basic/__init__.py::register`）。

**协议校验**：写完用 `dashboard_auth/base.py::assert_protocol_compliance(YourProvider)` 做单测，确保 ABC 全实现。

### 注意：CLI 本身没有"登录"概念

上面这套鉴权只保护 **Web/桌面/serve 的 HTTP 后端**。纯 CLI（`cli.py`）是本机进程，不存在登录门。如果你的贴牌产品主入口是 CLI 且要"登录后才能用"，那是新增需求：需要在 CLI 启动流程里加一道自定义校验（改 `cli.py` / `hermes_cli/main.py` 的启动路径），这属于内核改动，范围另评估。

## 五、落地清单（建议顺序）

1. **贴牌视觉**：加品牌皮肤（`skin_engine.py` 的 `_BUILTIN_SKINS` 或 `~/.hermes/skins/`），改 `DEFAULT_CONFIG["display"]["skin"]` 默认值；再改 Web/桌面的 Logo、标题、图标、favicon。
2. **锁 URL**：新建品牌供应商 profile → 改 `DEFAULT_CONFIG["model"]` 默认供应商 → 在 `run_agent.py` 落地处强制覆盖 `base_url`（含 delegation / auxiliary）→ 关掉 `/model`、setup、Web/桌面里改 URL 的入口。
3. **登录**：单密码用内置 `basic`；多用户登录注册则写一个 `DashboardAuthProvider` 插件 + 一个注册路由 + 前端注册页。
4. **上服务器**：Docker（`HERMES_DASHBOARD=1` + 映射 9119）或 `dashboard --host 0.0.0.0`，配 `HERMES_DASHBOARD_BASIC_AUTH_*` 三件套，前挂 HTTPS 反代。见第六节。
5. **验证**：
   - 测试脚本统一用 `scripts/run_tests.sh`（勿直接 `pytest`）。
   - TUI/Web/桌面前端：`ui-tui/`、`web/`、`apps/desktop/` 各自 `npm run typecheck && npm run lint && npm test`。
   - 手动核对：用户改 config.yaml 的 `model.base_url` 后，实际请求仍打到锁定 URL（抓包或看 `~/.hermes/logs/agent.log`）。
   - 服务器登录门：`curl -s http://IP:9119/api/status | jq '.auth_required, .auth_providers'`。

## 六、能不能上服务器？（能，官方就是自托管定位）

**误区澄清**：本机默认绑 `127.0.0.1` 只是"开箱即用不强制登录"的安全默认，**不代表只能本地**。Hermes 本身就是"开源自托管 AI Agent"，服务器部署是一等公民。

### 服务器上登录门何时开启

`web_server.py::should_require_auth`：绑定到**非 loopback**（`0.0.0.0` / 公网 IP / 局域网 IP）→ **强制**必须配置鉴权 provider（登录），否则**启动即失败**（fail-closed）。
`--insecure`（公网免登录）在 2026-06 安全加固后**已废弃为 no-op**——公网必须登录。这对"贴牌 + 登录"是正好的约束。

### 三种服务器部署方式

**A. 直接跑 dashboard（最简单）**
```bash
hermes dashboard --host 0.0.0.0 --port 9119 --no-open
```
需要的额外依赖：`uv pip install -e ".[web,pty]"`（web 用 FastAPI/Uvicorn，pty 用于浏览器内嵌 TUI 聊天）。

**B. Docker / docker-compose（最推荐，仓库已带 `docker-compose.yml`）**
官方镜像 `nousresearch/hermes-agent`。开启仪表盘：`HERMES_DASHBOARD=1`，映射端口：
```yaml
services:
  hermes:
    image: nousresearch/hermes-agent:latest
    restart: unless-stopped
    command: gateway run
    ports:
      - "8642:8642"   # 网关 OpenAI 兼容 API
      - "9119:9119"   # 仪表盘（HERMES_DASHBOARD=1 时可达）
    volumes:
      - ~/.hermes:/opt/data     # 全部状态在这里
    environment:
      - HERMES_DASHBOARD=1
      - HERMES_DASHBOARD_HOST=0.0.0.0
      - HERMES_DASHBOARD_BASIC_AUTH_USERNAME=admin
      - HERMES_DASHBOARD_BASIC_AUTH_PASSWORD=改成强密码
      - HERMES_DASHBOARD_BASIC_AUTH_SECRET=用 openssl rand -base64 32 生成
```
容器内 s6 会同时监管 gateway 与 dashboard，崩溃自动重启。镜像里 `/opt/hermes`（核心代码）是只读不可变，可变状态全在 `/opt/data`（= `~/.hermes`）。

**C. systemd 常驻**
```ini
[Service]
EnvironmentFile=%h/.hermes/.env
ExecStart=/path/to/venv/bin/python -m hermes_cli.main dashboard --host 0.0.0.0 --port 9119 --no-open
Restart=always
```
`.env` 里放 `HERMES_DASHBOARD_BASIC_AUTH_*` 三件套。

### 服务器登录的三种鉴权（选一，全走环境变量，不改代码）

| 方式 | 触发条件 | 适合场景 |
|------|----------|----------|
| 用户名密码 `basic` | `HERMES_DASHBOARD_BASIC_AUTH_USERNAME` + `_PASSWORD`（+ `_SECRET` 让会话跨重启有效） | 单账号、内网/VPN/自用，零基础设施 |
| Nous OAuth `nous` | `HERMES_DASHBOARD_OAUTH_CLIENT_ID` | 托管/公网 |
| 自建 OIDC `self_hosted` | `HERMES_DASHBOARD_OIDC_ISSUER` + `_CLIENT_ID` | 对接你自己的 IdP |

> **多用户登录注册**仍需自写 `DashboardAuthProvider` 插件（见第四节需求 2）——`basic` 只是单账号。

### 校验服务器登录门是否已开

```bash
curl -s http://你的服务器IP:9119/api/status | jq '.auth_required, .auth_providers'
# 期望： true   和   ["basic"]（或 nous / 你的自定义 provider）
```
- `auth_required: false` → 说明还是 loopback 绑定或没绑对 host。
- `true` 但没有 provider → 环境变量没加载，登录会失败。

### 上公网的安全清单

- 一定放在 **HTTPS 反代**（Caddy/nginx，Caddy 自动证书最省事）后面，别裸暴露。
- 仪表盘能读写 `.env`（含所有 API Key）与 config，务必强密码 + 防火墙只放行 443。
- 网关的 OpenAI 兼容 API（8642）默认关闭；要开需 `API_SERVER_ENABLED=true` + `API_SERVER_KEY`，否则别对公网开。
- 生产网关建议开工具循环熔断：`tool_loop_guardrails.hard_stop_enabled: true`。
- 别让两个 gateway 容器共用同一个 `~/.hermes`（会话/记忆非并发安全）。

### 上服务器后，用户怎么用？（客户端形态 = 决定"在哪执行"）

**能下载客户端本地执行——桌面 App 默认就是本地执行。** 但要先在两种形态里选一种，本质区别是 **Agent 的终端 / 文件 / 浏览器在谁的机器上跑**：

| | 形态一 · 瘦客户端 | 形态二 · 胖客户端 |
|---|---|---|
| 客户端 | 下载桌面 App（切 **Remote 模式**）或浏览器开 Web 仪表盘 | 下载桌面 App（默认 **Local 模式**）或 CLI/Docker |
| Agent 执行位置 | **在你的服务器**（服务器的 `~/.hermes`、终端、文件、浏览器） | **在用户本机**（用户的终端、文件、浏览器） |
| 客户端角色 | 只是聊天 UI，无运行时 | 装全套运行时，跑本地 `hermes serve` |
| 你的服务器提供 | 全部（模型 + Agent + 存储） | 只提供①锁定模型端点 ②登录/发 key |
| 锁 URL 效果 | **最彻底**：模型配置/key 从不下发客户端 | 客户端本地持有配置，见下方"坑" |
| 适合 | SaaS、集中管控、多用户（profile 隔离） | "我电脑上的 Agent"、省你算力、能碰用户本地文件 |

- 代码依据：桌面 App 的 `apps/desktop/src/app/settings/gateway-settings.tsx` 里 `type Mode = 'local' | 'remote'`；Remote 模式填服务器 URL + 登录 token/OAuth。远程模式下 Chat 会在**服务器**上以服务器的 `HERMES_HOME` 拉起 PTY 子进程执行。
- 形态二的模型入口：`gateway/platforms/api_server.py`（OpenAI 兼容 API，端口 8642，`API_SERVER_ENABLED=true` + `API_SERVER_KEY` 开启），或你自建代理。定制客户端 build 把 `model.base_url` 硬编码指向你的服务器。

#### 关键的坑：本地执行时"锁死 URL"只能锁在服务器端

项目是 **MIT 开源 + 纯 Python（非编译）**。形态二在本地执行时，配置就在用户机器上——**硬编码 URL 只挡得住普通用户，技术用户能翻出来改**。

- **真正可强制的锁在服务器端**：客户端只准打你的网关，网关要求**登录后签发的 per-user token 才响应**，没授权就调不到模型。这才是能落地的锁，别指望客户端本地文件改不了。
- 若"连本地执行都不许用户绕过"是硬要求 → 只能用**形态一（瘦客户端，服务器执行）**，配置永不下发。
- 混合可行：本地执行（碰用户文件）+ 模型走你服务器（服务器发 per-user key 做鉴权/计费）——功能上成立，但"URL 不可改"这条在开源客户端上无法 100% 保证，用服务器鉴权兜底。

### 目标方案 = 自建 Portal（本文重点：注册登录 + 本地端授权 + 积分）

**你的设想**：服务端是带注册/登录/积分的网站；用户下载本地端；本地端通过浏览器去服务端授权登录拿凭证；之后模型调用走服务端（可扣积分）。

**这正是 Nous 官方的 "Nous Portal" 架构**，代码里整套机制已现成，白标就是"建你自己的 Portal"。对应关系：

| 你的设想 | 代码里的现成实现 |
|---|---|
| 服务端网站，注册/登录 | Portal 服务器（`ProviderConfig.portal_base_url`，如 `portal.nousresearch.com`） |
| 本地端去服务端授权登录 | `auth_type="oauth_device_code"`（设备码 OAuth） |
| 拿授权凭证 | `hermes_cli/auth.py::_request_device_code` → `_poll_for_token` |
| 模型走服务端（锁 URL） | Portal 授权时下发 `inference_base_url` 代理端点，本地所有调用打它 |
| 积分 | `agent/account_usage.py`：`monthly_credits` / `credits_remaining` / `purchased_credits_remaining`，`/credits`、`/usage`，额度不足报 "Check credits or billing" |

**授权流程（设备码，本地端不碰用户密码）：**

1. 本地端 `POST 你的门户/api/oauth/device/code`（带 `client_id`、`scope`）→ 返回 `user_code`、`verification_uri`、`verification_uri_complete`、`expires_in`、`interval`。
2. 本地端提示："请浏览器打开 `verification_uri` 并输入 `user_code`"（或直接开 `verification_uri_complete`）。
3. 用户在**你的网站**完成注册/登录并批准授权（积分、套餐、计费都在这一侧）。
4. 本地端轮询 `POST 你的门户/api/oauth/token`（`grant_type=urn:ietf:params:oauth:grant-type:device_code` + `device_code` + `client_id`）；期间返回 `authorization_pending` / `slow_down` 就继续等，最终拿到 `access_token`（+ `refresh_token`）。
5. 本地端把 `model.base_url` 锁成 Portal 返回/约定的**代理推理端点**，每次请求带 `Authorization: Bearer <access_token>`。
6. **你的代理服务器**：验 token → 查该用户积分 → 转发给真实模型（OpenAI 兼容）→ 按用量扣积分 → 积分为 0 时拒绝（返回额度不足错误）。
7. token 过期时本地端用 `refresh_token` 静默续期（参考 `refresh_nous_oauth_pure` / `resolve_nous_runtime_credentials`）。

**为什么这套最契合你的需求：**

- **锁 URL 真正可强制**：推理端点在授权时由服务器下发，且必须带 per-user token；本地即使改了 `base_url` 也没用——没有效 token + 积分就调不动模型。这解决了"开源客户端本地配置可改"的根本问题（把强制点放到服务器）。
- 注册/登录/积分/计费**全在服务端**，本地端零信任、可开源分发。
- 本地端仍在**用户机器上执行**（终端/文件/浏览器），符合"下载本地端"的产品形态。

**你需要建/改的东西（三部分）：**

1. **服务端（你自己写，非本仓库）**：
   - 网站：注册、登录、积分/套餐、计费后台。
   - `POST /api/oauth/device/code` 与 `POST /api/oauth/token`（设备码 OAuth，协议照抄上面的字段）。
   - **推理代理**：OpenAI 兼容 `/v1/chat/completions` 等，入口验 `access_token` + 查/扣积分，再转发到真实模型供应商。
2. **本地端（改本仓库 = 你的 fork）**：
   - 新建品牌供应商插件 `plugins/model-providers/<brand>/` + 在 `hermes_cli/auth.py` 的 `PROVIDER_REGISTRY` 加一条 `ProviderConfig(auth_type="oauth_device_code", portal_base_url="你的门户", inference_base_url="你的代理", client_id=..., scope=...)`（照抄 `nous` 那条即可）。
   - 设备码登录/轮询/续期可**直接复用** `_request_device_code`、`_poll_for_token`、`refresh_nous_oauth_pure`、`resolve_nous_runtime_credentials` 的现成实现，只换 URL/字段。
   - 把品牌供应商设为 `DEFAULT_CONFIG["model"]` 默认，并按第三节做"运行时强制覆盖 base_url"兜底。
3. **积分展示（可选）**：接你服务端的账户接口，复用 `agent/account_usage.py` 的 `/credits`、`/usage` 渲染，本地端就能显示剩余积分。

> 一句话：**照着 `nous` 这个 provider + `hermes_cli/auth.py` 的设备码流程克隆一份指向你自己的门户**，服务端自建（注册/登录/积分/代理），就是你要的产品。CLI/桌面/网关全部自动走这条授权链，无需分别改。

### 服务端管理端（配模型 / URL / key / 积分定价 / 充值）——用现成的 one-api / new-api

**边界澄清（重要）**：
- 本仓库（Hermes）= **客户端/本地端**，**没有**多用户的"模型+积分+充值"管理端。
- `hermes dashboard` 只是**单实例配置台**（配这一台自己的模型/key），**不是**运营用户的 SaaS 后台。
- 你要的管理端属于**你自建的 Portal 服务器**，是独立产品——但**不必从零写**。

**这类"带计费的 LLM 网关"有成熟开源方案**，功能与你的需求一一对应：
- one-api：https://github.com/songquanpeng/one-api
- new-api（one-api 增强版，UI/支付更全）：https://github.com/QuantumNous/new-api

| 你要的管理端功能 | one-api / new-api 对应 |
|---|---|
| 配置各个模型 | "渠道(channel)"管理，加/删/改模型 |
| 模型的 URL 和 key | 每渠道填**上游 base_url + API key**（存服务端，永不下发客户端） |
| 模型的积分配置 | 每模型**倍率/定价**（按 token 或次数），分组价 |
| 充值 | **兑换码**批量生成导出 + 在线充值/支付、额度管理 |
| 用户管理（附带） | 注册/登录、用户额度、令牌、用量账单、限速、IP 白名单 |
| 统一出口（附带） | 输出 **OpenAI 兼容** `/v1/...`，正好做客户端锁定的 base_url |

**完整三层架构：**

1. **模型网关 + 管理端 + 积分/充值**：部署 one-api / new-api。管理员后台配模型渠道(url+key)、定价、发兑换码；用户在其自带前端注册/充值。
2. **授权衔接层（二选一）**：
   - **省事版**：用户在网关前端登录/充值后拿到 API token，粘贴进客户端。客户端 base_url 锁死到你的网关。**零额外开发**。
   - **体验版**（前述"浏览器授权登录"）：在网关前加一个薄的 **OAuth 设备码 shim**（照抄前一节的设备码接口），客户端点"登录"→浏览器→你的站点登录→发一个映射到该用户网关 token 的凭证。模型/定价/充值仍在 one-api 后台，shim 只管"发凭证"。
3. **客户端**：Hermes fork，`base_url` 锁到你的网关，请求带用户 token，可复用 `agent/account_usage.py` 显示剩余积分。

**数据流（模型调用）：**
```
客户端(带用户token) → 你的网关/api/v1/chat/completions
   → 网关验token → 查该用户积分 → 按模型定价预扣/计量
   → 用管理端里配的“上游 base_url + key”转发给真实模型商
   → 返回结果 → 扣积分 → 积分为0则拒绝(额度不足)
```

**关键收益**：上游真实 **URL 和 key 只存在管理端(服务器)，永不下发客户端**——把"锁 URL/锁 key"从"客户端不可改"（开源客户端做不到）变成"服务器强制"（做得到）。客户端手里只有一个可吊销的用户 token。

> 结论：**管理端 = one-api / new-api**（配模型/URL/key/定价/充值/用户，一站式）；你只需要（可选）加一层设备码 shim 做"浏览器授权登录"，再把客户端 base_url 锁到网关。

#### 纯二开 Hermes 做服务端管理端（支付 / 微信登录 / 邮箱登录 / 对话+图片模型）——可行

**结论：可以，一套 Hermes 代码库就能做，不必引入 one-api / new-api。** Hermes 的仪表盘后端本身就是 **FastAPI + React** 的可二开 Web 应用，已提供以下现成脚手架：

| 你要的 | Hermes 里现成的二开点 |
|---|---|
| 邮箱 / 微信 / 密码登录 | `hermes_cli/dashboard_auth/` 的 `DashboardAuthProvider` ABC——写子类即可；内置 `basic` 密码 provider 可直接抄 |
| 后台页面（模型/用户/订单/充值） | **仪表盘插件**：后端出一个 FastAPI `router`，被 `hermes_cli/web_server.py::_mount_plugin_api_routes()` 自动挂到 `/api/plugins/<name>/`；前端加 Tab / 页面（`web/src/pages/` 或插件主题） |
| 充值 / 计费 | 已有 `hermes_cli/nous_billing.py`——它是**计费 API 的客户端**（`/api/billing/state`、`/charge`、`/auto-top-up`、`/charge/{id}` 全套契约）。二开指向你自己的计费服务，或照它的契约实现服务端 |
| 模型配置 / 对话模型 | 模型 config（provider / base_url / key）+ 供应商 profile |
| 图片模型 | `plugins/image_gen/` 图像供应商 |
| 积分显示（客户端侧） | `agent/account_usage.py`（`/credits`、`/usage` 渲染，含 `monthly_credits` / `credits_remaining`） |

**登录方式怎么加（邮箱/微信）**：在 `plugins/dashboard_auth/` 下按 `basic` 的写法各加一个 provider——
- 邮箱：`supports_password=True` + 邮箱验证码校验，或标准 OAuth。
- 微信：`start_login` 返回微信扫码授权 URL，`complete_login` 用 code 换 openid → 建/取用户 → 返回 `Session`。
- 全部经 `ctx.register_dashboard_auth_provider(...)` 注册，登录页自动渲染。

**后台管理页怎么加**：做成一个仪表盘插件，`api` 文件暴露一个 FastAPI `router`（自动挂 `/api/plugins/<brand>/...`），前端加对应管理 Tab。用来管：用户、模型目录、每模型定价、订单/充值记录、支付回调（微信支付/支付宝/易支付）。

##### 必须自己写的部分（诚实告知，换哪个底座都要写）

1. **计量代理（带扣积分的原始模型转发）** —— 关键坑：
   `gateway/platforms/api_server.py` 的 `/v1/chat/completions` **是"服务端起一个 AIAgent 跑 Agent"，不是原始模型透传代理**。
   你的客户端在本地跑 Agent、把"模型调用"打到服务器计费，需要一个**原始透传代理**：验用户 token → 查积分 → 用后台配的 `上游 url+key` 转发到真实模型商 → 扣积分 → 零则拒绝。
   这块 Hermes 没有，**自己写**（就写成上面的仪表盘插件 FastAPI 路由，如 `/api/plugins/<brand>/v1/chat/completions`）。
2. **多用户账号 + 积分账本 + 模型目录/定价** —— 净新增：Hermes 给你 FastAPI/React/登录框架/计费客户端，但账户表、积分流水、模型定价、订单/支付业务逻辑要自己建（DB + 路由 + UI）。

##### 两套代码库（都在 Hermes 生态内）

1. **服务端**：Hermes fork，作为 `hermes dashboard` / `serve` 跑在服务器上。加：登录 providers（邮箱/微信/密码）+ 管理后台插件（用户/模型/定价/充值/支付）+ 计量代理路由 +（可选）设备码授权接口。
2. **客户端**：Hermes fork，分发给用户。贴牌 skin + `base_url` 锁到你服务器的计量代理 + 授权拿 token + 复用 `account_usage` 显示积分。

两者用"**OpenAI 兼容协议 + 用户 token**"对接；上游真实 URL/key 只存服务端，**永不下发客户端**。

> 诚实取舍：纯二开 Hermes 完全可行，语言栈统一（Python + React）、品牌统一、可控；代价是"计量代理 + 多用户计费"这块业务逻辑要从零写（这恰是 one-api/new-api 已解决的部分——若想省这块可用它做模型网关，但你已明确不引入，则在 Hermes 里自建即可）。

## 八、关键红线（来自项目规范 AGENTS.md）

- **不要破坏 per-conversation 提示词缓存**：不要在对话中途改系统提示、换工具集、重载记忆。
- **非机密配置放 `config.yaml`，不要新增 `HERMES_*` 环境变量**；`.env` 只放密钥。
- **插件不得改内核文件**（`run_agent.py`/`cli.py`/`gateway/run.py` 等）。需要新能力就扩展通用插件接口。锁 URL 的"运行时强制覆盖"属于对**你自己 fork 的内核**的定制，不是插件行为——想象成可维护的话，尽量用集中常量 + 少量改动点。
- 路径一律 `get_hermes_home()` / `display_hermes_home()`，不要硬编码 `~/.hermes`。
