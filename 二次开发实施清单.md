## 一、四件套架构

```
┌── ① 客户端（贴牌 Hermes 桌面端 apps/desktop，分发给用户，本地执行）──────────────────────────┐
│  品牌：名称 / logo / 图标 / skin                                                              │
│  登录门：点“登录” → 跳系统浏览器到 Casdoor → 登完授权码回传客户端（PKCE）                        │
│  品牌 provider「Working Plan / vLLM」：base_url 锁到 ③ 的计量代理（也可保留用户自带 key 的 provider）│
│  之后本地跑 Agent（用户的终端/文件/浏览器），模型请求带 token/key 打向 ③                          │
└───────┬─────────────────────────────────────────────────────┬────────────────────────────────┘
        │ ①浏览器 OAuth 授权码                                  │ OpenAI 兼容 + Authorization
        ▼                                                       ▼
┌── ② 认证 = Casdoor（开源自部署）──────────┐   ┌── ③ 业务后台（你自建/二开）+ 计量代理 ─────────────┐
│  登录 / 注册 / 邀请码                        │   │  用户：与 Casdoor 用户关联、积分余额、状态          │
│  手机 / 邮箱 / 微信 / 社交登录               │   │  模型配置：目录(name/type/上游url/上游key/定价)      │
│  发 access_token（OIDC/OAuth2）              │   │  支付：充值下单 + 回调验签 + 加积分                 │
│  自带用户管理后台                            │   │  积分：流水 / 扣费 / 余额                           │
│  给「客户端」「业务后台」各建一个 OAuth 应用  │   │  计量代理：验token→查积分→按“上游url+key”转发→扣费  │
└──────────────┬───────────────────────────┘   │  DB：users/credits/model_catalog/orders           │
               │ 校验 token / 拉用户            │  上游真实 url+key 只存这里，永不下发客户端 ──► 真实模型商 │
               └───────────────────────────────┴──────────────────────────────────────────────────┘

┌── ④ 官网（静态站）─────────────────────────────────────────────────────────────────────────┐
│  介绍 + 下载页（macOS .dmg / Windows .exe / Linux .AppImage）+ 跳注册（去 Casdoor）           │
└──────────────────────────────────────────────────────────────────────────────────────────────┘
```

### 登录链路（浏览器授权码 + PKCE，跟竞品一致）
```
客户端生成 code_verifier/code_challenge → 打开系统浏览器：
  {casdoor}/login/oauth/authorize?client_id=..&response_type=code&scope=openid profile
     &code_challenge=..&code_challenge_method=S256&redirect_uri={回调}
用户在 Casdoor 登录/注册（可填邀请码）→ Casdoor 302 到 redirect_uri?code=..&state=..
回调有两种落地：
  (a) loopback：客户端本地起 http://127.0.0.1:<port>/callback 直接收 code
  (b) 中转页 + deeplink：{oauth}/oauth/success 收 code 后用 brand:// 深链拉回 App（竞品图4 的做法）
客户端用 code + code_verifier 换 token（PKCE，无需在客户端存密钥）
换到 token 后：调 ③ 业务后台注册/绑定用户 → 领取“该用户的推理 key” → 写进品牌 provider 的 api_key
之后模型请求：base_url=③计量代理，Authorization: Bearer <该用户key>
```

## 四、③ 业务后台（自建/二开）：用户 · 模型配置 · 支付 · 积分 + 计量代理

> **已定：路线 A｜纯二开 Hermes 自建**（本节以下全按 A 施工）。
> 作为 `hermes dashboard` 插件——FastAPI router 自动挂到 `/api/plugins/<brand>_admin/`
> （`web_server.py::_mount_plugin_api_routes` [复用]），后台前端页放插件里或 `web/src/pages/`。
> 控制力最强、复用 Hermes 的鉴权/挂载；计量代理 + 多用户计费 + 后台 UI 全部自写。
> （路线 B「Casdoor + one-api/new-api」已放弃，仅作对比记录。）

### 4.1 数据库（新建，路线 A）
- `users`：id、casdoor_user_id、角色、状态、（可缓存邮箱/手机）
- `credits_ledger`：user_id、变动、原因（充值/消费/兑换/赠送）、时间
- `model_catalog`：name、type(chat/image)、upstream_base_url、upstream_api_key(加密)、price/倍率、enabled、是否限免
- `orders`：user_id、金额、支付渠道、状态、第三方单号
- `api_keys`：user_id、推理 key（登录后发给客户端当 provider key）、过期/吊销

### 4.2 后台功能（新建）
| 模块 | 内容 | 说明 |
|---|---|---|
| 用户管理 | 列表 / 封禁 / 调积分 / 关联 Casdoor 用户 | 登录态校验 Casdoor token |
| 模型配置 | 增删改：name、类型(对话/图片)、上游 url+key、定价/倍率、限免开关 | 上游真实 key 只存这里（加密）|
| 支付充值 | 下单 + 回调验签（微信支付/支付宝/易支付）+ 加积分 | 兑换码可选 |
| 积分管理 | 余额、流水、消费明细 | |
| 后台前端 | 管理 UI（**React + Vite + Radix UI + shadcn/ui**，单独 SPA）| 不并入 `web/`（那边是 `@nous-research/ui`，非 shadcn）；SPA 调本插件 `/api/plugins/<brand>_admin/*` 接口 |

### 4.3 计量代理（核心新建，路线 A）
| 任务 | 落点 | 类型 | 说明 |
|---|---|---|---|
| 透传代理 | `plugins/<brand>_admin/<api>.py` 里加路由，如 `/api/plugins/<brand>/v1/chat/completions` | [新建] | 验 key/token → 查积分 → 用 `model_catalog` 的上游 url+key 转发 → 统计用量 → 扣积分 → 零则拒绝 |
| SSE 流式 | 同上 | [新建] | 支持 chat completions stream 透传 |
| 模型列表 | `/v1/models` | [新建] | 返回目录（只给名字/定价，不含 key），客户端 `/model` 选择器据此列出 |
| 别用 api_server | `gateway/platforms/api_server.py` | [勿改] | 它是“服务端起 AIAgent 跑 Agent”，不是透传代理，不适合当模型端点 |

### 4.4 客户端账户接口（新建）
| 接口 | 说明 |
|---|---|
| `POST /account/bind` | 客户端带 Casdoor token 来 → 建/取用户 → 返回该用户推理 key |
| `GET  /account/credits` | 返回余额/用量（供桌面端积分显示）|

## 六、施工里程碑（建议顺序）

1. **M1 打通登录**：部署 Casdoor + 建客户端 OAuth 应用；客户端接授权码+PKCE 登录门（先能登录、拿到 token）。目标：点“登录”→浏览器→登完回到 App。
2. **M2 品牌 provider + 锁 URL + 领 key**：新增品牌 provider（base_url 锁 ③）；登录后向 ③ 领推理 key 写入 provider；三层锁死 base_url。目标：改配置也改不掉端点。
3. **M3 计量代理**：③ 出 `/v1/chat/completions` 透传 + `/v1/models`；先纯转发跑通，再加“查积分/扣费/余额为 0 拒绝”。目标：能对话且扣积分。
4. **M4 业务后台 + 支付**：用户/模型配置/积分页 + 支付渠道回调 + 充值。目标：可视化配模型 url+key+定价、用户自助充值消费闭环。
5. **M5 贴牌收尾 + 官网**：skin/名称/图标 + 三平台打包 + 官网下载页 + Casdoor 登录页品牌化。

## 七、待定决策（动手前定）

1. ~~业务后台路线~~ ✅ **已定：路线 A（纯二开 Hermes 自建）**。
2. **回调方式**：loopback（`127.0.0.1` 本地收 code，最稳）还是 deeplink（`brand://`，体验跟竞品图4一致，但要注册系统协议）。
3. **推理鉴权**：客户端调计量代理用「Casdoor access_token」还是「后台发的专用推理 key」。推荐后者（与 Casdoor 会话解耦、可单独吊销/限额）。
4. **限免 vs 纯付费**：是否要“Working Plan / vLLM 限免额度”那种免费档（竞品有）。

## 八、验证（每个里程碑都要过）

- 后端测试：`scripts/run_tests.sh`（勿直接 `pytest`）。
- 前端：`apps/desktop`、`web/`、`ui-tui/` 各自 `npm run typecheck && npm run lint && npm test`。
- 登录核验：点登录→浏览器 Casdoor→回 App，`~/.hermes` 下有 token/凭据。
- 锁 URL 核验：手改 `config.yaml` 的 `model.base_url` 后抓包/看 `~/.hermes/logs/agent.log`，请求仍打你的计量代理。
- 扣费核验：余额为 0 时调用被拒；充值后可用。
- 计量代理核验：`curl {代理}/v1/models` 只返回名字不含 key；上游 key 不出现在客户端任何配置/日志。
